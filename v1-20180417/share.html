一.简述

优点：
1.面对不同分辨率设备灵活性强
2.能够快捷解决多设备显示适应问题
缺点：
1.兼容各种设备工作量大，效率低下
2.代码累赘，会出现隐藏无用的元素，加载时间加长
3.其实这是一种折衷性质的设计解决方案，多方面因素影响而达不到最佳效果
4.一定程度上改变了网站原有的布局结构，会出现用户混淆的情况
常见问题：
1. 客户是否想支持互联网用户增长最迅猛的市场？如果想，那响应式方法就很适合。
2. 客户是否想要最简洁、快速，且易于维护的代码？如果想，那响应式方法就很适合。
3. 客户能否理解用户体验可以且本应该根据浏览器不同而不同？如果可以理解，那响应式方法就很适合。
4. 客户是否要求设计效果在所有浏览器中都保持一致，包括IE 8以及更低版本？如果是，响应式设计就不适合。
5. 该网站的当前或预期客户中，是否有百分之七十以上的人可能使用Internet Explorer 8或者更低版本？如果是，则响应式设计不适合。

在预算允许的情况下，一个完全定制的“移动”版网站比响应式设计更适合。澄清一下，我将那些完全专注移动平台、为移动设备用户提供不同内容/体验的解决方案称之为“移动网站”。提倡响应式设计方法的人，不会都认为响应式设计在任何情况下都可以替代“移动网站”。

三.写法

两个关键点：
1. 是如何在不修改Dom结构的前提下调整布局。
2. 是如何判断屏幕分辨率并应用对应的CSS。
3. 并且以上两点都应该不依赖于JS（特殊情况涉及功能的除外）。

首先，要在html的head加入

<meta name="viewport" content="width=320px, initial-scale=1, maximum-scale=1,minimum-scale=1, user-scalable=no">

然后，开始处理css

1. 直接在css里写

    使用最小宽度(即大于768px)
    @media screen and (min-width:768px){

       //你的样式放在这里...
       body {}

    }

    or:
    使用最大宽度(即大于979px)
    @media screen and （max-width:979px){

       //你的样式放在这里...
       body {}

    }

    or:
    最小宽度和最大宽度混合使用(即大于768p小于979px)
    @media (min-width:768px) and (max-width:979px){

       //你的样式放在这里...
       body {}

    }

2. 可以单独写个css，然后在html的头文件里做判断，不同的设备引用不同的样式

    <link rel="stylesheet" media="screen and (min-device-width : 320px) and (max-device-width: 480px)" href="iphone.css" />
    <link rel="stylesheet" media="screen and (min-device-width : 768px) and (max-device-width : 1024px)" href="ipad.css" />

四.举个简单的例子

/*宽度小于1024px*/
@media screen and (min-width:1023px){
    body { background:white; }
}
/*宽度小于980px*/
@media screen and (min-width:979px){
    body { background:pink; }
}
/*宽度小于800px*/
@media screen and (min-width:799px){
    body { background:yellow; }
}
/*宽度小于768px*/
@media screen and (min-width:767px){
    body { background:orange; }
}
/*宽度小于768px*/
@media screen and (min-width:767px){
    body { background:black; }
}

    body { background:orange; }
}
